# CLAUDE.md — Engineering Standards for This Project

This file governs all code written in this repository. Follow every rule here without exception.

---

## 1. Package Management — `uv` only

- **Never use `pip`, `pip install`, or `python -m pip`.**
- Add runtime dependencies with `uv add <package>`.
- Add dev/test dependencies with `uv add --dev <package>`.
- Run scripts and tools with `uv run <command>` (e.g. `uv run pytest`, `uv run python src/main.py`).
- `pyproject.toml` is the single source of truth for dependencies. Do not edit `requirements.txt` for new deps; it is kept only for legacy reference.

---

## 2. Data Models — Pydantic v2

- **All data models must be `pydantic.BaseModel` subclasses.** Never use `@dataclass` or plain dicts for domain objects.
- Use `model_validator`, `field_validator`, and `computed_field` where appropriate.
- Use strict types: `int`, `str`, `bool`, `list[X]`, `dict[str, X]`. Avoid `Any`.
- Default to `model_config = ConfigDict(frozen=True)` for value objects (nodes, connections, fragments). Use mutable models only when mutation is genuinely needed.
- Never use `dict()` to serialise models — use `model.model_dump()`.

---

## 3. LLM Integration — Pydantic AI + Google Vertex AI

- **Use `pydantic-ai` as the sole agent/LLM orchestration layer.** Do not call `google-cloud-aiplatform` or `vertexai` SDK directly.
- Declare agents with `pydantic_ai.Agent`, specifying:
  - `model='google-vertex:gemini-2.0-flash'` (or configurable via env, default to this)
  - `result_type=<SomePydanticModel>` so all LLM responses are validated automatically
  - `system_prompt` as a module-level string constant in `src/prompts.py`
- Authenticate via Application Default Credentials (`gcloud auth application-default login`). Do not hard-code credentials.
- LLM calls that could fail must use pydantic-ai's built-in retry mechanism (`retries=` param on `Agent`). Do not wrap in manual try/except retry loops.
- All structured outputs from the LLM must be Pydantic models. Never parse raw LLM text with regex or string splitting.

### Vertex AI env vars (read from `.env`)
```
VERTEX_PROJECT=my-gcp-project
VERTEX_LOCATION=us-central1
VERTEX_MODEL=google-vertex:gemini-2.0-flash
LLM_MAX_RETRIES=3
```

## 4. SQL Dialect — MSSQL / T-SQL only

- **All generated SQL must be valid Microsoft SQL Server (T-SQL) syntax.**
- Never emit PostgreSQL, MySQL, SQLite, or ANSI-only constructs unless they are also valid T-SQL.
- Specific rules:
  - Quote identifiers with square brackets: `[schema].[table]`, `[column_name]`
  - Use `TOP N` (not `LIMIT N`) for row limiting
  - Use `NVARCHAR`, `DATETIME2`, `BIT` for string/datetime/boolean types
  - Use `ISNULL()` (not `COALESCE()` where avoidable, but `COALESCE` is valid T-SQL)
  - Use `GETDATE()` / `GETUTCDATE()` for current timestamp
  - Use `TRY_CAST` / `TRY_CONVERT` for safe type conversion
  - Window functions: `ROW_NUMBER() OVER (PARTITION BY ... ORDER BY ...)`
  - CTEs: `WITH cte_name AS (SELECT ...) SELECT ...` — every output is a CTE chain
  - String concatenation: `+` operator (not `||`)
  - Boolean literals: `1` / `0` (not `TRUE` / `FALSE`)

## 5. Progress Tracking — Project Plan Checkboxes

- The authoritative task list is `docs/PROJECT_PLAN.md`.
- **When you complete a file or module listed in the checklist, immediately update that checkbox from `[ ]` to `[x]`.**
- When all items in a phase are checked, mark the phase header with `**[DONE]**`.
- Do not mark items done speculatively — only after the code is written and (where applicable) tests pass.

---

## 6. Phase Documentation

- **When a phase is fully complete, write a documentation file at `docs/phase<N>_<name>.md`.**
- The documentation must cover:
  - Purpose of the phase
  - Module(s) written and their public API (key functions/classes with signatures)
  - Key design decisions and why they were made
  - How to extend or modify the phase
  - Example usage (code snippet)
- Do not write phase docs speculatively. Write them only when all checkboxes for the phase are ticked.

---

## 7. Workflow Documentation Output

- Every conversion run produces **two output files** alongside each other:
  - `<name>.sql` — the generated T-SQL CTE script
  - `<name>_docs.md` — human-readable workflow documentation (see Phase 8 in the project plan)
- The docs file describes both the original Alteryx workflow (what each tool did, data flow) and the generated SQL (what each CTE does, how they map to the original tools).
- The docs file is generated by the documentation agent (`src/doc_agent.py`) using pydantic-ai, not assembled from templates.

---

## 8. Code Style

- Line length: 88 characters (ruff default).
- All source files must pass `uv run ruff check src/ tests/` with zero errors.
- Format with `uv run ruff format src/ tests/` before finalising any file.
- Imports: standard library → third-party → local, separated by blank lines (ruff enforces this).
- No unused imports, no commented-out code blocks, no `print()` debugging (use `rich` for output).
- Type annotations on every function signature. No bare `Any` unless truly unavoidable.
- Module-level docstrings on every `src/` file explaining the module's responsibility.

---

## 9. Testing

- Tests live in `tests/`. Run with `uv run pytest`.
- Fixtures (sample `.yxmd` XML, expected SQL) live in `tests/fixtures/`.
- LLM calls in tests must be mocked with `pytest-mock` — never make real Vertex AI calls in unit tests.
- Integration tests (real LLM calls) are marked `@pytest.mark.integration` and excluded from the default run.
- Aim for deterministic translators to be 100% covered without mocks; LLM-assisted paths need mock coverage.
